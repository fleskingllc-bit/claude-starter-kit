Chapter 1

ローカルにプロジェクトが溜まる問題

Q Claude Codeを使っていたら、Macのホームディレクトリにプロジェクトフォルダが30個近く溜まってきたんだけど。これ、みんなどうしてるの？

これ、Claude Codeに限らず、AIコーディングツール（Cursor、Copilot等）を使う人が全員ぶつかる課題です。

調べてみると、コミュニティの共通見解は明確でした。

GitHub private repository（無料・無制限）にpushするのが最もコスパの良いバックアップ。
Q リポジトリが30個になるのは多すぎない？ パスが崩れたり、プログラムが壊れたりしない？

全く問題ないです。GitHub無料プランでprivate repo数は無制限。プロのエンジニアは数十〜数百個のリポジトリを持つのが普通です。

それから、pushしてもローカルのフォルダ構成は一切変わりません。プログラムの動作も何も影響を受けません。

Q あ、そうなんだ。じゃあGitHubは「コードの金庫」みたいなもので、動かすのは引き続きローカルのファイルってことか。

その通りです。GitHubにpushした後も、動かすのは今まで通りローカルのファイルです。

Macが壊れたら新しいMacでGitHubから git clone して復元する。それだけの話です。

Chapter 2

ビルドとデプロイ — コードはどこで動いているのか

Q GitHubにpushしたコードは、結局どこで動くの？ ローカル？ クラウド？

実は2種類あります。

実行場所仕組み例
💻ローカル（自分のMac）macOSの定期実行機能で動かすデータ収集、ファイル整理、OCR処理
☁️クラウド（サーバー）GitHubにpush → 自動デプロイ管理画面、Webアプリ、社内Wiki
  • クラウドで動いているもの → 自分のMacが閉じていてもユーザーがアクセスできる
  • ローカルで動いているもの → Macが起動していないと止まる
Q なるほど。コードをGitHubにプッシュして、それがクラウドのサーバーにデプロイされて、動くのはサーバー上で動く。これが「クラウドで動いている」ということだよね。

はい、その理解で正確です。流れとしてはこうなります：

① ローカルでコードを書く
→
② GitHubにpush
→
③ サーバーがビルド＆デプロイ
→
④ ユーザーがアクセス
Q ビルドって何？ ビルドの後にデプロイなんだよね。デプロイは「送る」みたいな意味？
  • ビルド = 人間が書いたコードを、ブラウザが読める形に変換すること。たくさんのファイルが数個のファイルにまとまり、圧縮される
  • デプロイ = ビルドされたファイルをサーバーに送って公開すること

Netlifyのようなサービスはこの「ビルド→デプロイ」を自動でやってくれるので、開発者はGitHubにpushするだけで済みます。

Q ユーザーがデプロイされた、ビルドされたものを見た場合に、元のコードがわからないよね。ビルドされた状態だから結果しかわからない。その裏側のコードがどうなっているのか、これを「ソースコード」と言うんだよね。

完璧な理解です。

名前中身
📝ソースコード人間が書いた元のコード
📦ビルド成果物ブラウザ用に変換・圧縮されたもの

ビルド成果物は変数名も短縮されて1行にまとめられるので、人間が読んでもまず解読できません。

だからこそ、元のソースコードをGitHubに保管しておく必要があるんです。

Chapter 3

Git操作の基本 — commit / push / pull

Q コミットって何？

「この時点のコードを記録する」操作です。ゲームのセーブポイントみたいなものです。

作業1: ログイン画面を作った → コミット💾（セーブ） 作業2: メール一覧を作った → コミット💾（セーブ） 作業3: バグを直した → コミット💾（セーブ）

各コミットにはメッセージをつけます。例えば「認証トークンの期限切れ対応を修正」のように。

これにより「いつ、何を変えたか」の履歴が全部残ります。問題が起きたら特定のコミットまで巻き戻すこともできます。

Q コミットとプッシュの関係は？
コードを書く
→
commit（ローカルに記録）
→
push（GitHubに送る）

commitはローカルだけの操作です。何回commitしても、pushするまでGitHubには反映されません。

こまめにcommitして、区切りの良いところでpush。これが基本です。
Q プッシュは自分でGitHubに変更を送ること。プルは何？

逆です。GitHubの変更をローカルに取ってくることです。

別のPCやチームメンバーがGitHubにpushした変更を、自分のPCに反映するときに git pull します。1人で開発している場合はあまり使う場面はありません。

Chapter 4

プルリクエストとコンフリクト

Q プルリクエスト（PR）って何？ プッシュとどう違うの？
  • プッシュ = 自分で直接コードを変更する
  • プルリクエスト = 「この変更を取り込んでください」という提案

チーム開発で使われる流れ：

  1. メインのコードとは別の「ブランチ（枝）」で変更を作る
  2. 「この変更をメインに取り込んで（pullして）ほしい」とリクエストを出す
  3. チームメンバーがレビューして、OKなら取り込む
つまりプッシュが「自分で直接変える」なら、プルリクは「レビューが入る」という違いです。
Q ローカルにプルしてきた後に変更を加えて、プッシュまたはプルリクするときに、本体の方でも更新がかかって、差分に矛盾が生まれた場合、どうなるの？

それがコンフリクト（競合）と呼ばれるものです。

✅ 矛盾がない場合
Gitが自動でマージ（合体）してくれる。Aさんが1行目を、Bさんが50行目を変えた場合 → 別の場所なので自動で両方反映。
❌ 矛盾がある場合
同じ行を別々に変えた場合、Gitは「どっちが正しいかわからない」と止まる。人間が判断する必要がある。

コンフリクトが発生すると、コードにはこんなマークが付きます：

<<<<<<< あなたの変更 const color = 'blue' ======= const color = 'red' >>>>>>> 相手の変更
Q なるほど、プルリクするときに事前にコンフリクトを検知してレビューするってことか。開発ブランチを切ってプルした後に開発して、もう一回プルして確認してからプルリクないしプッシュするっていうこともあるよね。それでもコンフリクトは起きる？ その場合、個人で解決するの？

はい、もちろんあります。むしろ一般的です。

よくある流れ：

  1. ブランチを切って開発する
  2. 開発中にmain（本体）が進む（別の変更がマージされた等）
  3. git pull origin main で最新を取り込もうとする
  4. コンフリクト発生
  5. 自分で該当箇所を見て、どちらを採用するか判断して修正
  6. 修正をcommitして続行

コンフリクトの解決自体は「このファイルのこの行、どっちの書き方が正しい？」という判断です。コードを理解している人なら誰でもできます。

Q みんながその「作業前にpullする」習慣が必要なんだね。それ意外と難しいな。リテラシーがない人もいるわけだし。コードレビューもClaude Codeがやっていくことになりそうだ。

まさにそうだと思います。

実際、現時点でもClaude Codeがcommitメッセージを書いてpushまでやりますし、レビュー機能も備えています。Gitのリテラシーがなくても安全に開発できる方向に確実に進んでいます。

Chapter 5

複数人での開発とデータベース

Q 複数人がClaude Codeで同じプロジェクトを進める場合、誰かがGitHubでプロジェクトを立ち上げて、コントリビューター（貢献者）としてみんながプルリクしていくっていう流れになる？

はい、基本的にはその流れです。

  1. 誰かがGitHubにリポジトリを作る
  2. 各メンバーがclone（コピー）して、ブランチを切って開発
  3. プルリクで変更を提案 → レビュー → マージ
Q この場合、データベースをどうするかが気になるな。コードはGitHubで共有できるけど、データはどうするんだろう。

鋭いポイントです。コードはGitHubで共有できるが、データベースはそうはいかない。ここが難しいところです。

よくある3パターン：

構成本番DB開発DB向いている規模
スプレッドシート型Google Sheets 1つを共用同じものを使う〜数百件、少人数
ローカルDB分離型クラウドDB 1つ各自のPCにテスト用DBスタートアップ
完全分離型本番DBステージングDB + 開発DB大規模チーム
Q ローカルDBって、ソフトウェアとして何に乗っかってるの？ Google Sheetsでもいいし、単純なCSVデータでもいいわけか。

その通りです。CSVでもGoogle Sheetsでも「データベース」にはなります。用途によって適切なものが変わるだけです。

種類例特徴
スプレッドシートGoogle Sheets人間が見やすい、小規模向け
ファイルDBSQLiteファイル1個で完結、サーバー不要
サーバーDBPostgreSQL, MySQL大規模・高速・同時アクセスに強い
クラウドDBSupabase, PlanetScaleサーバーDBのクラウド版、管理不要
小規模なら Google Sheetsを本番用と開発用の2つに分けるだけで十分。SQLiteやPostgreSQLへの移行が必要になるのは、データが数万件を超えたり、複雑な検索が必要になってからです。
Chapter 6

.envファイルと秘密情報の管理

Q .envファイルってよく出てくるけど、何のファイルなの？

APIキーやパスワードなどの秘密情報を書いておくファイルです。

.env GOOGLE_CLIENT_SECRET=xxxxxxxxxxxxx API_KEY=sk-ant-xxxxxxxxxxxxx

こういう情報をコードに直接書くと危険なので、.envという別ファイルに分けて書き、プログラムがそこから読み込む仕組みです。

Q これ、GitHubにpushしないファイルなんだよね？ ということはローカルにしかない。なくなったら消えちゃう情報ってこと？

鋭い。はい、.envはGitHubには上げません。

.gitignoreという「GitHubに送らないファイルリスト」に入れて除外します。これは開発者の間では常識中の常識で、フレームワーク（Vite、Next.js等）が新規プロジェクト作成時に自動で設定してくれることも多いです。

Q じゃあ本番のサーバーはどこから秘密情報を読んでるの？

本番とローカルで設定場所が完全に分かれています。

場所設定方法用途
☁️クラウドサーバー（本番）サーバーの管理画面で直接設定本番用の秘密情報
💻ローカル（開発）.envファイル開発用の秘密情報

だから.envが消えても本番は影響ゼロです。ローカルで開発するときに再設定が面倒、という程度の話。しかもサーバーの管理画面に同じ値が入っているので、最悪そこから確認して復元できます。

Q なるほど。GitHubに上げていく.envファイルはローカルの方から上げるわけだから……いや待って、.envはGitHubには上げないんだったよね。

そうです。整理するとこうなります：

GitHub → .envは含まれない（.gitignoreで除外）
サーバー → 管理画面で環境変数を直接設定（本番用）
ローカル → .envファイルで設定（開発用）

.envはあくまでローカル専用の設定ファイルです。

Q 本番用と開発用でDBを分けるっていうのも、この.envを変えるだけでできるの？

その通りです。.envの中のデータベースのIDを開発用に差し替えるだけで、同じコードが開発用DBを参照するようになります。

コードは一切変えずに、接続先だけ切り替える。これが .env の便利なところです。
Chapter 7

launchdとは

Q launchdってシェルスクリプトみたいなもの？

少し違います。

名前役割
📋シェルスクリプト「何をするか」の手順書（例：ファイルを整理する手順）
⏰launchd「いつ実行するか」を管理するmacOSの仕組み（例：毎月1日の朝9時に実行）

Windowsでいう「タスクスケジューラ」、スマホでいう「アラーム」と同じ役割です。macOSに最初から入っている機能で、設定ファイル（.plist）を書くだけで使えます。
